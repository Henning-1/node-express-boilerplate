# Runbook: Open an incident Linear tracking ticket

Every declared production incident gets exactly one Linear ticket,
opened via `create_linear_issue`. The ticket is the post-incident
review's single anchor — every artifact (Slack thread, GitHub issue,
PR, commit, runbook, ADR) hangs off it.

## Source-trail discipline (read before drafting)

While diagnosing, keep a running list of **every URL** any ctx tool
returns — `diagnose_incident`, `find_similar_incidents`,
`get_runbook`, `search_context`, `search_all`, `get_pr_diff`, etc.
You will cite each URL **verbatim** in the `## Sources` section of
the ticket. Do not summarize URLs away. If a `## Sources` subsection
ends up empty, write `_(none)_` — do not omit the heading.

## Title

One line, present tense, names the system + the incident date + the
bug class. Embed the channel-name date stamp so the ticket and
channel sort together:

- "Auth 401s on prod — incident-2026-05-09 — JWT expiry regression"
- "Checkout 500s — incident-2025-09-14 — order serializer canary gap"
- "TechDocs build failing — incident-2024-11-03 — generator OOM"

## Defaults

- `priority`: 2 (high) for sev-2; bump to 1 for sev-1.
- `state`: "In Progress" at creation. Move to "Done" only after the
  fix is shipped *and* the post-incident review is filed.
- `team_key`: CLI (cline-hackathon).

## Description (markdown) — required sections

The description must be markdown and include all six H2 sections
below plus the H2 `## Generated by` footer, in this order:

### `## Diagnosis`

2–4 sentences. Most likely root cause + explicit confidence level +
the reasoning behind that confidence ("Confidence: high because
<2–3 reasons: matches past incident X, runbook section Y points
here, blame trail lands on Z>").

### `## Suspect change`

- **PR:** PR title (linked PR url)
- **File:** `<file>:<line>`
- **Commit SHA:** `<sha>`
- **Author:** <name returned by `git_blame`>
- **Framing vs effect:** one line — gap between the PR's stated
  description (title + body) and what the diff actually did. If the
  PR was honest about its effect, say so explicitly. Silence here
  is ambiguous.

### `## Owner`

`<github handle>` per CODEOWNERS rule `<pattern>`. This is the
person to page.

### `## Mitigation`

A concrete revert command using the commit SHA (`git revert <sha>`)
plus the expected ETA. Quote any constraint from the runbook clause
the change violated.

### `## Sources`

Every URL the diagnosis was grounded in, cited verbatim as
`[label](url)`. Group as below. If a subsection is empty, write
`_(none)_`:

  **Slack threads**
  - `[<channel> · <one-line excerpt>](url)`
  - … (one bullet per Slack URL)

  **Linear (past incidents)**
  - `[<identifier> — <title>](url)`
  - … (one bullet per Linear URL)

  **GitHub (runbooks · ADRs · code)**
  - `[<file_path> §<section>](url)` — include line ranges when known
  - … (one bullet per GitHub URL)

### `## Generated by`

`` `ctx-mcp · diagnose_incident` · <UTC timestamp> ``

## After creating the ticket — Slack synthesis post

Call `post_to_slack` into the incident channel created earlier. Use
Slack mrkdwn (`*bold*`, `_italic_`, `<url|label>`):

  *Diagnosis* — <one-liner with confidence>
  *Suspect:* <PR linked> · `<file>:<line>` · commit `<sha>`
  *Owner:* <handle to page>
  *Mitigation:* <revert command>
  *Tracking:* <CLI-N linked>
  *Sources:* the most relevant 2–3 URLs as `<url|label>` — keep it
             tight; the full source trail lives in the Linear ticket.

  _:robot_face: ctx-mcp synthesis_

## Final reply to the on-call requester

Three lines, exactly:

- Linear: `<CLI-N>` `<url>`
- Slack: `<permalink>`
- Sources cited: `<integer count of unique URLs in the ## Sources
